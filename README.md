# Multi-Tracker Dashboard (Offline-First Greenfield Build)

A local, offline-first personal dashboard for tracking progress across multiple domains simultaneously (e.g. course syllabi, running logs, fitness parameters, habits, etc.) using a consistent, registry-based tracker abstraction.

## Features

- **Registry-Based Multi-Tracking**: Create and manage multiple trackers (e.g. "AAT Level 3" course syllabus and a "Running Log" side by side).
- **Course Tracker Workflows**: Group syllabus topics dynamically under collapsible headers, log study reviews, write concept notes, and track test scores.
- **Micro-Heuristic Study Signals**: Automatically labels topics with learning heuristics (e.g. *Friction*, *Needs retrieval*, *Brittle*, *Ready test*, *Efficient*) based on study time, reviews count, and pass/fail thresholds.
- **Global Study Session Timers**: Start study sessions with per-topic timers. Displays ticking duration metrics and active banner controls across the application.
- **Offline-First & Local**: No accounts, no servers. All data is saved inside the browser's `localStorage`.
- **Universal JSON Import Pipeline**: Import topics or dated activity logs from pasted JSON generated by an AI assistant from screenshot templates or descriptions.
- **Robust Storage Engine**: Built-in corruption recovery to guarantee data integrity.
- **Elegant Aesthetic**: Curated color-palette theme featuring paper background, gold highlights, and ink-text accents supporting both Light and Dark modes.

## Getting Started

To run the application, open the root `index.html` file directly in any modern web browser. No compilation, bundlers, or installation steps are required.

```bash
# Double click index.html or open via a local server:
npx serve .
```

## Storage Recovery Pattern

The application handles data updates by creating backup versions of keys during saving:
1. When saving to `key`, the current value is backed up to `key:prev`.
2. If `key` fails to load or parse, it automatically falls back to restoring from `key:prev` and quarantines the corrupted string in `key:corrupt` for later recovery.
3. If both fail, the entry is quarantined, and the state initializes safely to prevent blank screens or crashes.
