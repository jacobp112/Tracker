## 1. Product summary
A fully local, no-login, offline-first personal dashboard for tracking progress across multiple domains at once — study/courses (e.g. AAT modules), physical activity (running, weight training), and any other custom pursuit — using a consistent "tracker" abstraction, with data entered either manually or via pasted JSON generated by an AI from a screenshot or description.

**Scope boundary, confirmed:** no accounts, no backend, no cross-device sync. All data lives in the browser's `localStorage` on whichever device the app is opened on. The trade-off (no sync between phone and laptop) is accepted.

**Deployment target:** a static site with zero required build step — plain HTML/CSS/JS with ES modules, openable directly as a local file or deployable to GitHub Pages for easy access from any single device's browser.

---

## 2. What to take from the legacy dashboard (study it, then reimplement)
Read `/reference/legacy-dashboard/` before building. Specifically study and reimplement (cleanly, not copy-pasted) these proven patterns:
- **The tracker/topic data model**: status, confidence, review history, per-item timers, and the "study signal" heuristics (friction/needs-retrieval/brittle-fluency/ready-test/efficient) that turn raw time+confidence data into actionable labels.
- **The JSON-tag-and-reimport workflow**: how the legacy app tags an outgoing test with an id, lets the user run it through an AI, and matches the returned JSON back to the right record on import (`testId` → `pendingMarking`). This is the direct template for "screenshot → AI → JSON → import" for any tracker type.
- **The Import modal pattern**: paste → preview → validate → apply, with a visible error state and a preview state, never applying unvalidated data directly.
- **The visual language**: the paper/gold/ink CSS custom-property theme with a `body.dark-theme` override block, `Fraunces`/`Newsreader` typography, and the card/panel styling. Reuse the aesthetic; don't reuse the file.
- **Corruption-recovery storage pattern**: primary key + `:prev` fallback + `:corrupt` quarantine key, so a bad save never silently destroys data.

Do not carry over: the hardcoded single-syllabus `TOPICS` array, or any maths-specific logic — that becomes user-generated data in the new model.

## 3. Architecture

### 3.1 Repository structure
```
/
  index.html
  /src
    /core
      storage.js          // localStorage get/set + corruption-recovery, key namespacing per tracker
      trackerRegistry.js   // create/list/archive/delete trackers
      schema.js            // FieldSchema definitions + validation
      importPipeline.js    // template generation, validate, preview, apply
    /trackers
      courseTracker.js     // status/confidence/reviewHistory/timers/study-signal logic
      logTracker.js        // dated entries against a flexible field schema, aggregates
    /ui
      main.js               // app entry, wiring, tab switching
      home.js
      trackerList.js
      courseView.js
      logView.js
      chartsView.js
      importModal.js
      createTrackerModal.js
    /styles
      theme.css             // CSS variables + dark theme, ported from legacy aesthetic
      layout.css
      components.css
  /reference
    /legacy-dashboard/      // old files, read-only inspiration, never imported by new code
  /docs
    README.md
    ARCHITECTURE.md
```

### 3.2 Core data model
```
Tracker {
  id: string
  name: string
  type: "course" | "log"
  schema: FieldSchema[]
  createdAt: string
  archived: boolean
}

FieldSchema {
  key: string
  label: string
  kind: "text" | "number" | "date" | "select"
  unit?: string          // e.g. "km", "kg", "min"
  options?: string[]      // for "select" kind
}
```
- `course` trackers additionally hold a list of topics/sections (name, status, confidence, reviewHistory[], studySeconds), directly generalizing the legacy topic model.
- `log` trackers hold a list of dated entries, each entry being a plain object matching the tracker's `schema`.

### 3.3 Storage
- Each tracker gets its own namespaced key: `tracker:<id>:meta`, `tracker:<id>:entries`.
- A single `trackers:index` key holds the registry (list of tracker ids/names/types) so the app can enumerate trackers without loading all their data.
- Every write goes through the corruption-recovery pattern from Section 2 (primary + `:prev` + `:corrupt` quarantine), implemented once in `storage.js` and reused by every tracker — not reimplemented per tracker type.

### 3.4 Universal JSON import
- Any tracker can generate an example JSON payload + a matching "paste this into an AI" prompt block from its own `schema`.
- Import modal is one shared component (`importModal.js`) parameterized by tracker id + schema, not a separate modal per tracker type.
- Preview always shown before apply; apply is the only step that writes to storage.

## 4. Definition of Ready (a story may not be started until:)
- The exact `FieldSchema` for any tracker type involved is written out in the story — no fields invented mid-implementation.
- The story names which modules it creates or touches, and confirms no logic is being duplicated across `courseTracker.js` / `logTracker.js` that belongs in `storage.js` or `importPipeline.js` instead.
- Acceptance criteria are written as a testable checklist (see stories below).

## 5. Definition of Done (a story may not be merged until:)
- No console errors on load, tracker creation, import, and tab switching, in both light and dark theme.
- New UI uses only the CSS variables defined in `theme.css` — no new hardcoded hex colors.
- Every new storage-writing code path goes through `storage.js`'s corruption-recovery functions, not raw `localStorage` calls.
- Pure-logic modules (`schema.js` validation, `storage.js` merge logic, study-signal heuristics in `courseTracker.js`) have basic unit tests using a minimal, dependency-free test runner (a simple `assert`-based script run via `node`, no test framework dependency required).
- `README.md` is updated to describe any new tracker type or workflow added.
- Opened via `index.html` directly (no build step) and confirmed working, in addition to whatever deploy target is used.

## 6. Epics and user stories

### Epic A — Project scaffold + core engine (build first)
**A1.** As a user, I open `index.html` and see an empty but fully functional shell: tab bar (Home / Trackers), light/dark theme toggle, no errors, no trackers yet.
- [ ] `storage.js` implements get/set with corruption-recovery, covered by unit tests.
- [ ] `trackerRegistry.js` can create, list, archive, and delete a tracker, covered by unit tests.
- [ ] Theme variables and dark-mode toggle ported from legacy aesthetic and working.

**A2.** As a user, the universal import modal exists and works end-to-end against a dummy tracker/schema (preview → validate → apply), proving the pipeline before any real tracker type depends on it.

### Epic B — Course/syllabus tracker (priority 1 — needed for AAT L3 in September)
**B1.** As a user, I can create a new course tracker by pasting a syllabus/module handbook.
- [ ] Dashboard generates a "paste this into an AI" prompt specifying the exact JSON shape needed (`section`, `name`, `reference` fields).
- [ ] Pasting the returned JSON creates a tracker with topics, each supporting status, confidence, review history, and a timer, reimplementing (not copying) the legacy study-signal logic (friction/needs-retrieval/brittle-fluency/ready-test/efficient).

**B2.** As a user, I can run multiple course trackers side by side (e.g. "AAT Level 3" and any other course) and switch between them without losing state in either.

### Epic C — Activity/log trackers (priority 2)
**C1.** As a user, I can create a "Running" log tracker by describing the fields I want (distance, duration, pace, date), and get a JSON template + AI prompt for converting a screenshot into an importable entry.

**C2.** As a user, imported entries appear in a dated list/table for that tracker with simple aggregates (e.g. total distance this week).

**C3.** As a user, I can view a simple line chart for any numeric field in a log tracker over time (e.g. pace over time). Use a lightweight, dependency-free SVG chart or a single CDN-loaded chart library — agent's choice, documented in `ARCHITECTURE.md`.

### Epic D — Home aggregation, tracker management, deploy
**D1.** As a user, my Home view shows a one-line snapshot per active tracker (e.g. "Maths: 3 topics need retrieval · Running: 2 runs this week").

**D2.** As a user, I can rename, archive, or delete a tracker from a management panel, with a confirmation step before delete.

**D3.** As a user, the app is deployed to GitHub Pages (or documented equivalent) so I can open it from my phone browser, with clear documentation that data does not sync between devices.

## 7. Git / GitHub workflow (agent must follow exactly)
- **Branch naming**: `feat/<short-description>`, `fix/<short-description>`, `docs/<short-description>`.
- **Commits**: Conventional Commits — `feat: add course tracker creation flow`, `fix: correct corruption-recovery fallback order`, `docs: document log tracker schema format`.
- **One story (or tightly related pair) per PR.** Epic A must be merged and manually verified before Epic B/C work begins.
- **PR description template:**
  ```
  ## Summary
  What this PR does and which user story it implements.

  ## Manual QA performed
  - [ ] No console errors (light + dark theme)
  - [ ] Opens via index.html with no build step
  - [ ] Unit tests pass (if touching core/ or trackers/)
  - [ ] README/ARCHITECTURE updated if relevant

  ## Follow-ups
  Anything intentionally deferred.
  ```
- **Docs**: `README.md` (usage) and `ARCHITECTURE.md` (structure/decisions) are kept current as part of each story, not as separate cleanup work.
- **Merging**: squash-merge into `main` once the PR checklist is satisfied. No force-pushes over `main`.
